# Architecture Requirements — W138 V1

## 1. Framing

This is a **requirements document derived from empirical survivors**, not a claim that a coordination-native architecture exists or has been built. The long-term programme goal is: *"Can CoordPy turn the mechanism stack it has been testing — memory, routing, verification, iterative feedback, and auditable coordination — into a coordination-native neural architecture that outperforms plain transformers on hard, contamination-resistant tasks at matched compute?"* W138 is a **blocker-closing step** toward that architecture, not the architecture itself. The requirements below are the conditions any such architecture MUST structurally satisfy to survive the reproduced empirical blocker chain **W120–W137** and to reproduce the two confirmed same-budget wins **W89/W105**. Each requirement traces to specific milestones and primary-source literature; none is generic ML advice. If a proposed architecture violates any requirement below, the chain predicts it will FAIL at matched compute on a contamination-resistant field.

## 2. Requirements

### R1 — The architecture must generate NEW correct trajectories, not re-select from the i.i.d. sample pool.
**Evidence.** W125 ($0 re-routing capped: A1 = oracle pass@K is an upper bound a blind selector cannot beat; only new correct trajectories help), W128 (role-diverse search LIFTS the generation ceiling, reaching a program the i.i.d. pool missed), W129 (committed ≤ pool ceiling < earn bar *regardless of selector* ⇒ cap is in generation), W130 (attacking the generator cracked exactly 1 pool-dead problem). Self-consistency is inert when the answer is not already in the pool (arXiv:2203.11171).
**Architectural implication.** A generation module whose output distribution is *not* a fixed i.i.d. resample of one head — e.g. a coordination policy that conditions generation on prior failed trajectories/feedback to produce candidates outside the base sampling support. Re-ranking alone is structurally insufficient.

### R2 — The architecture must contain an internal verifier STRONGER than the generator head, with access to a hidden/execution-grounded signal.
**Evidence.** W128 (selection-capped: no hidden-test oracle to pick the pool-only win; abstains/mis-commits), W129 (a complexity-aware verifier *correctly chose* the hidden-correct program citing complexity — public+derived signal alone was non-discriminating; pawnshop A0≡B1 on all public signal). Intrinsic self-correction does not help verifiable reasoning and can hurt; gains require an EXTERNAL verifier whose strength EXCEEDS the generator, and small models specifically need strong verifiers (arXiv:2310.01798; arXiv:2404.17140).
**Architectural implication.** An internal verifier module that is provably stronger than the generator on the target distribution (e.g. an execution/complexity-grounded checker, not a same-strength self-explanation), wired as a distinct head/sub-module — not the generator scoring its own output.

### R3 — The architecture must route feedback FORM by failure mode.
**Evidence.** W130 (counterexample-to-rewrite cracked the one hidden-edge problem doubleup; complexity-gated handoff, family coach, budget router added 0), W133 (the EW2 complexity witness earned +12.12pp vs A1 / +6.06 vs B0 on complexity modes while the EW1 counterexample witness added +0.00 on wrong-algorithm/edge modes — feedback form is mode-specific), W136 (an I/O-repair directive cracked 3/3 traps that algorithm feedback in any form could not). Established ranking: failing-test/counterexample > execution/stderr > intrinsic self-explanation (arXiv:2304.05128; arXiv:2401.08500).
**Architectural implication.** A routing module that classifies the failure mode and selects the matching feedback channel (counterexample vs complexity witness vs I/O-format repair vs edge case) — a single uniform feedback form is dominated.

### R4 — The architecture must contain an internal difficulty/solvability estimator and act only in the informative band.
**Evidence.** W132 (A0=A1=76.67%, B=80.00%, B−A1=+3.33pp clean FAIL on a saturated field), W137 (repaired field is BIMODAL at 70B: every template's A1∈{0,1}; 12/17 saturated, 4/17 dead, 1/17 in-band ⇒ no mechanism-sensitive headroom), W89/W105 (earned precisely because HumanEval(+) has a MIDDLE BAND, A1≈84%, the model partially solves). Selection/consistency carry information only at p∈(0,1) (arXiv:2203.11171); an item is informative only near p≈0.5 (IRT θ=b; saturated/dead items are zero-information — arXiv:2407.12844; arXiv:2511.04689).
**Architectural implication.** An internal per-item difficulty/uncertainty estimator that gates the coordination machinery ON only where p∈(0,1), and falls back to single-shot generation on saturated/dead items (where coordination cost buys nothing).

### R5 — The architecture must spend compute adaptively across the difficulty band (matched-compute discipline).
**Evidence.** W125–W130 all held budget at exactly same-budget K=5 and still failed by SPAN/headroom — uniform compute on a bimodal field wastes the budget on saturated and dead items. W137's surviving band is 1/17 of templates; a same-budget mechanism must concentrate compute there. The two wins (W89/W105) are *same-budget* sequential reflexion, i.e. compute spent inside the band, not added overall.
**Architectural implication.** A budget-allocation/router module that reallocates the fixed compute toward in-band items (R4) — coordination must beat pass@K *at matched total compute*, so the lever is allocation, not extra calls.

### R6 — The architecture must be robust to I/O / surface-format confounds (separate parsing capability from algorithmic capability).
**Evidence.** W136 (the apparent "wrong-algorithm ceiling" of W132–W135 was an I/O-FORMAT CONFOUND — the model wrote correct algorithms but misparsed whitespace-flattened input; quadruple-confirmed: standard-I/O A0 one-shots 3/3, identical DP passes 7/7 with split-parsing and fails 7/7 per-line), W137 (parser-neutral kernel + HC1 gate make the confound machine-impossible to recur).
**Architectural implication.** An input-normalisation / parsing sub-module decoupled from the reasoning core, so that a verifier's "wrong" verdict reflects algorithm error, not surface misparse. A coordination net must not conflate I/O failure with capability failure.

### R7 — The architecture must address WRONG-ALGORITHM (capability-bound) failures with a generation capability, not feedback alone.
**Evidence.** W130 (dominant WRONG_ALGORITHM_ADMISSIBLE failures — familyvisits/foodprocessor/hilbertshedgemaze — cracked by NO arm; named technique ≠ correct algorithm = capability-bound), W135 (oracle-derived PROSE solution-structure did NOT lift the wrong-algorithm field at 70B; structure-unliftable, not merely counterexample-unliftable), W127 (family scaffolds failed on the resistant field). Feedback-form ranking applies only once the generator *can* produce the right algorithm (arXiv:2304.05128).
**Architectural implication.** A generation module with genuine algorithmic capability (not a re-skinned prompt scaffold) — coordination cannot manufacture an algorithm the generator fundamentally cannot produce; either the base capability is sufficient (R1) or the win is unreachable on that mode.

### R8 — The architecture's mechanism must transfer from EXPOSED dev to RESISTANT eval (no memorisation leak).
**Evidence.** W127 (family scaffolds earned weakly on EXPOSED problems but FAILED on RESISTANT ones — dev signal did not transfer to the non-memorizable field), W113 (within-model exposed +10pp collapsed to +0.00 resistant at the same Maverick scale), the W109/W110 contamination dissociations. Surface re-skinning does not generalise; only continuous structural variation does (arXiv:2406.04229 CLRS-Text; arXiv:2509.26553 FuncBenchGen).
**Architectural implication.** The coordination mechanism must be validated on a *construction-resistant* distribution; any module whose gain depends on exposure/memorisation (R8 violation) is disqualified. Earn must be demonstrated where the answer cannot have been seen.

### R9 — The architecture must be evaluated on a multi-MODE, multi-FAMILY band (span discipline), not a single mode.
**Evidence.** W133/W134 (the complexity witness is REAL but SINGLE-MODE / single-family ⇒ §7a SPAN gate fails), W137 (the complexity witness still earns +33pp on surviving complexity cells but the field spans only 1 mode / ~3 families ⇒ SPAN failure, $0 frontier). A win on one mode is not a coordination win; it is a single-trick witness.
**Architectural implication.** The architecture's advantage must hold across ≥2 fixable modes and ≥2 families simultaneously — i.e. the routing/verification stack (R2,R3) must be *general* across modes, not tuned to one. This is the empirical bar W138 constructs a field to test.

### R10 — The architecture must reproduce the W89/W105 MIDDLE-BAND conditions, not assume them.
**Evidence.** W89 (Llama-3.3-70B × HumanEval, same-budget sequential reflexion B−A1=+5.56pp, multi-seed, 1st retirement), W105 (same model × HumanEval+, +7.00pp, 2nd retirement) — both EXPOSED HumanEval-family at 70B with a partial-solve band A1≈84%. The entire W120–W137 chain is the record of that band being *absent* on resistant/clean fields at 70B (W132 saturated, W137 bimodal).
**Architectural implication.** The architecture's evaluation must guarantee a populated p∈(0,1) band by construction (per R4); the two confirmed wins are existence proofs *conditional on the band*, so the architecture must either (a) target models/fields where the band exists, or (b) internally create solvable intermediate sub-problems. Coordination superiority is band-conditional, not unconditional.

### R11 — The architecture's coordination must be auditable / falsifiable (no fake-different mechanism).
**Evidence.** Across W125–W137 every mechanism was gated by a structural *fake-different* test that repeatedly BIT (relabeled-reflexion negative controls and trivial-bridge controls classified FAKE_DIFFERENT; substrate-trust controllers killed as not code-consensus-applicable in W128/W130; boilerplate-leakage false positives caught and corrected in W127/W130). The whole programme's credibility rests on this audit layer.
**Architectural implication.** An auditable coordination protocol with built-in negative/positive controls and leakage guards — the "auditable coordination" term in the goal is a hard structural requirement, not decoration; any module must pass a fake-different and a no-leakage check.

### R12 — The architecture must use continuous structural difficulty knobs to instantiate the band, not surface re-skins.
**Evidence.** W137 (per-template difficulty is BIMODAL — discrete templates landed at A1∈{0,1}, no in-between, so a discrete slate cannot populate the band), W132 (saturated discrete field). Continuous knobs (input size, recursion/dependency depth, distractor count) produce smooth difficulty; surface re-skinning does not (arXiv:2406.04229; arXiv:2509.26553).
**Architectural implication.** The difficulty estimator/curriculum (R4) must be driven by *continuous* structural parameters so that a p∈(0,1) band is reachable by tuning a knob — discrete template minting demonstrably collapses to {0,1} at 70B.

## 3. What W138 specifically tests of these requirements

W138 is the first empirical probe of the **core triad — R1 (new-trajectory generation), R2 (verifier stronger than generator), R3/R9 (feedback-form routing across ≥2 modes)** — under the band condition **R4/R10/R12**. Concretely, W138 tests: *can a **CONSTRUCTED band** (continuous structural knobs, R12) with **≥2 fixable modes** (R9) let a **same-budget, verifier-routed mechanism** (R2,R3,R5) **beat pass@K across scales** (R1, matched-compute)?* It is a deliberately *favourable* test of whether — once the band exists by construction (closing the W132/W137 absence) and ≥2 modes are present (closing the W133/W134/W137 single-mode SPAN cap) — verifier strength, feedback-form routing, and new-trajectory generation can jointly clear the earn bar. A PASS is evidence the triad is architecturally on-track; a FAIL localises which of R1/R2/R3 is the residual blocker. W138 does **not** build the architecture; it validates the *preconditions* the architecture's modules must satisfy.

## 4. Explicitly OUT OF SCOPE for W138 (deferred to the architecture build)

- **Differentiable / trained modules.** No weight-bearing internal verifier, learned memory, or trained routing head (the W136 learned-memory line was killed as random-until-trained on synthetic data; only weightless controllers are usable here). W138 stays $0-trainable; the *trained* realisation of R1/R2 is an architecture-build task.
- **Frontier-model certification and supply.** No resolution of the W123 supply cap / W131 no-certifiable-stronger-model gate (decision CID 258b6ed7, {KNOWN:1, UNKNOWN:4}); W138 uses the constructed/known-model field, not a new frontier model.
- **End-to-end coordination-native net.** No integrated architecture, no version bump, no PyPI, no `coordpy/__init__.py` change. W138 closes the band/mode *preconditions* only; assembling R1–R12 into a single coordination-native transformer alternative is the deferred architecture milestone.
